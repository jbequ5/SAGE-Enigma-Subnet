## Abstract

The Operations Subsystem is the conductor layer of SAGE. It manages the execution of Enigma Machine (EM) instances at scale, from a single local run to a full swarm of parallel operators, while collecting both normal fragments and rich operations telemetry.

Its core responsibilities are:
- Detect available compute resources and decide how many EM instances to run.
- Launch, monitor, and manage EM instances intelligently.
- Assign contracts/tasks to instances in a way that maximizes data quality and diversity.
- Collect operations telemetry (resource usage, launch parameters, success rates, cost/time per fragment, stall frequency, etc.).
- Feed both normal fragments and operations data into Solve → Strategy → Synapse for hierarchical learning.

This subsystem turns Phase 1 (solo data generation) into a self-improving parallel data factory and prepares SAGE for large-scale miner participation.

## 1. Spectrum of Operations Options

The Operations Subsystem supports a continuous spectrum of deployment modes:

**Level 1 – Single Local EM Instance**  
- One EM process running on the user’s machine.  
- Used for testing, debugging, or small-scale data generation.  
- No orchestration needed — direct launch via `python -m enigma_machine`.

**Level 2 – Local Parallel EM Instances (Recommended Phase 1)**  
- Multiple EM instances (4–20+) running concurrently on one machine.  
- Orchestrator uses Python `multiprocessing`, `subprocess`, or Docker to manage a pool of workers.  
- ResourceMonitor detects available CPU cores, GPU VRAM, and memory to set safe concurrency (e.g., never exceed 80% VRAM across all instances).

**Level 3 – Containerized Local Swarm**  
- Each EM instance runs in its own Docker container.  
- EM Operations agent uses Docker SDK to spin up/down containers dynamically.  
- Easy isolation, reproducible environments, and simple scaling.

**Level 4 – Cloud / Distributed Swarm**  
- EM Operations agent launches instances on cloud VMs, spot instances, or container platforms (AWS, GCP, Hetzner, RunPod).  
- Supports heterogeneous hardware (different GPU types, CPU-only nodes).  
- Cost-aware scheduling becomes active here.

**Level 5 – Full Agentic Operator Swarm (Future)**  
- The Operations layer itself becomes a small Arbos-style swarm with specialized roles (Resource Scout, Task Scheduler, Failure Analyst, Experiment Designer).  
- Hierarchical intelligence: the swarm learns to optimize orchestration policies over time.

The Operations Subsystem can operate at any level and transition smoothly between them.

## 2. Operations Level Intelligence

The Operations Subsystem is not just a launcher — it is an intelligent conductor that learns how to run EM instances more effectively.

**Key Intelligence Capabilities:**

- **Resource-Aware Concurrency Control**  
  \[
  N = \min\left( \left\lfloor \frac{\text{available_VRAM}}{\text{per_instance_VRAM_estimate}} \right\rfloor, \text{max_safe_concurrency} \right)
  \]

- **Task Assignment and Diversity Control**  
  The orchestrator assigns contracts/tasks from a queue, deliberately varying seeds, prompt templates, or contract slices to maximize heterogeneity in the generated fragments.

- **Stall Detection and Recovery**  
  Monitors each instance for stalls (no fragment output for > T minutes). Applies recovery actions: re-run with different seed, reduce complexity, or kill and re-assign.

- **Operations Telemetry Collection**  
  Every run logs:
  - Launch parameters (contract hash, seed, hardware assigned)
  - Resource usage (peak VRAM, CPU, duration)
  - Outcome metrics (fragments produced, final EFS, stall count)
  - Cost/time per high-signal fragment

- **Hierarchical Learning Loop**  
  Operations telemetry is sent to Synapse as a separate dataset. Synapse treats orchestration choices as actions and learns policies such as:
  - “Symbolic-heavy contracts run best on high-VRAM GPUs”
  - “Certain contract templates produce 3× more high-signal fragments”
  - “When stall rate > X, reduce concurrency by 1”

This creates a true outer learning loop that improves how SAGE generates data, independent of the inner EM solving loop.

## 3. Synapse Connections

The Operations Subsystem has two direct connections to Synapse:

**Connection 1 – Normal Fragment Flow**  
Fragments produced by any EM instance flow through Solve → Strategy → Synapse exactly as before. This is unchanged.

**Connection 2 – Operations Telemetry Feed**  
- Operations logs are written to a dedicated JSONL stream.  
- A lightweight Operations Logger ingests them and feeds them into Synapse as additional training data.  
- Synapse meta-RL loop treats operations telemetry as extra objectives or features, allowing it to tune:
  - Concurrency policies
  - Task assignment strategies
  - Hardware preferences (future hardware-aware routing)
  - Recovery thresholds

This dual feed creates hierarchical self-improvement: inner EMs improve solving intelligence, outer Operations improves data-generation efficiency.

## 4. Data Flow Summary

EM Instances → Solve (gating + 60/40 scoring)  
EM Instances → Operations Logger (telemetry)  
Solve → Strategy (ranked fragments)  
Strategy → Synapse (ranked + enriched intelligence)  
Operations Logger → Synapse (operations telemetry for meta-RL)  
Synapse → EM Instances (latest global scoring approximation at startup)

## 5. Attack Vectors and Mitigations

- Resource exhaustion attacks → ResourceMonitor + safety margins  
- Task queue poisoning → provenance validation on all assigned contracts  
- Telemetry gaming → cross-validation against actual fragment outcomes and EFS lifts

All operations are logged with full provenance and can be audited by the AHE.

## 6. Meta-Tuning Interaction

Synapse’s global meta-RL loop tunes both inner EM parameters (via the 60/40 weights, θ_dynamic, etc.) and outer Operations parameters (concurrency rules, task assignment policies, recovery thresholds). Local TPE tuning in each EM instance remains unchanged.

This creates a clean hierarchical meta-learning system.

## Why the Operations Subsystem Matters

The Operations Subsystem turns SAGE from a single-run tool into a scalable, self-improving data factory. It lets a solo builder (or small team) generate massive volumes of high-signal fragments in Phase 1 while simultaneously learning how to orchestrate EM instances more efficiently. By collecting both solving fragments and operations telemetry, it closes the outer learning loop that makes the entire SAGE flywheel accelerate faster than any individual miner could achieve alone.

From one local EM instance to a full swarm of intelligent operators, the Operations Subsystem provides the practical execution layer that makes the “People’s Intelligence Layer” real and scalable.
